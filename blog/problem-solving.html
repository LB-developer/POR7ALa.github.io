Problem Solving.

Talk about a time you were blocked on a simple problem.

What was the problem?

I was working through an exercise that should have been producing the correct answer, I constructed a simple for loop that on each iteration added or removed an item from an array depending on a condition.

What problem solving techniques did you use?

Checking the relevant documentation (.push() in this case), using pseudocode to make sure I understood the logic, creating a checklist of what should be happening and checking the relevant code.

How did you feel throughout the process?

Like an idiot; It's like I thought I had followed a recipe perfectly, but how was something going wrong if I could go over the checklist again and again, and find nothing.

What did you learn?

Look for typos


Talk about a time you elegantly solved a problem

What was the problem?

I was trying to construct a code box for showing coding examples in my blog.

What problem solving techniques did you use?

Googling! There were many different answers, mostly on StackOverflow (a popular developer forum).

How did you feel throughout the process?

Frustrated. The current implentation I had was extremely manual, (highlighting each word by a given colour to represent code highlighting) and was taking far too much time to achieve for such a small thing.

What did you learn?

Borrow from others. I ended up deleting all of my codebox styling and instead imported a popular codebox tool called prism.js that automatically styles your code for you. Although it does add dependency issues in the long run, it let's me focus on the content of the blogs themselves rather than the labourious styling I was implementing.


Reflect on how confident you feel using each of these problem-soliving techniques/processes

Psuedocode

I am confident using pseudocode either as a beginning i.e. "What do I want this section to do, overall" or/aswell as working alongside the code generation process.

Trying something

I prefer "trying something" for smaller tasks, it's quite easy to keep track of what I need something to do when the task is small, especially when the solution is the answer to a question, additionally, it feels like reinforced learning to watch, and be able to debug why something is going wrong in real-time, though I think this will become a less preferable method as I become more familiar/comfortable with coding in general.

Rubber ducky method

Rubber ducky is still something I'm practicing, it's so easy to fall back into my own head. I usually pair any rubber ducky method with some type of checklist/journaling of the problem and use the writing as checkpoints for explaining

Reading error messages

Reading error messages alone depends on the error itself, some seem very complicated and I need to google the error code to even understand what it's saying, however, something that I am confident with is using the ability to see where the error is; It contextualizes the errors in a way that just seeing the code itself would not provide.

Console.logging

I use console.log() extensively, getting feedback from the code itself, where we are, what is happening, or rather, what we want to be happening; It is one of the first things I do when implenting a new piece of code

Googling

I think in general I'm quite confident in being able to condense my problems into their root issues (the foundational reason something is not working), though I think I need to learn more about using specific search parameters like "site:" and using quotes. I believe those kinds of things will allow me to find relevant answers faster.

Asking your peers for help

A popular 'mantra' I have heard is that "if you are struggling with something, someone else has too", Asking peers for help is the easiest way to take advantage of this mantra and I'm very confident about leveraging the knowledge of others.

Asking coaches for help

I do struggle with this in a way. I have no problem reaching out and asking for help, but I am always looking to work through a problem to learn as much as possible, rather than get the answer correct and have "completed" the problem. I feel like, if I overcome my lack of understanding then I will learn more, but it is a fine line between pushing through and wasting time.

Improving your process with reflection

I am now starting to try and understand my interpretation of problems at the beginning, which I can refer to at the end and compare, if something wasn't what I expected, why? I find it a lot easier said that done though, especially with long/detailed projects, understanding your understanding of something is almost another task in itself, which, when frustrated with working through problems, is easy to cast aside.
